The Jarvis doctrine — what separates this from ChatGPT with voice

It acts, not just talks

Jarvis executes real tasks: sends emails, books meetings, writes and runs code, controls your computer, places orders. Every response ends in a real-world change if needed.

It knows you deeply

Persistent multi-layer memory: episodic (what happened), semantic (what you know), procedural (how you work), social (who you know). It never asks what you told it before.

It anticipates, not just responds

Background agents monitor your calendar, health metrics, finances, weather, and goals — surfacing what matters before you ask. Proactive, not reactive.

It sees everything

Camera, screen capture, document upload — Jarvis has eyes. It reads your screen mid-task, identifies objects, reads documents, and understands spatial context.

It runs anywhere

Local-first architecture: wake word, STT, and light tasks run on-device with zero latency and full privacy. Heavy tasks escalate to cloud. Works offline for core functions.

It learns your patterns

Reinforcement from feedback. Jarvis learns which email tone you prefer, what time you like reminders, how you structure your day — adapting silently over time.

System philosophy

Jarvis is built as a multi-agent orchestration system — not one monolithic model. A fast router (Haiku) classifies intent and assembles context. Specialist agents handle tools. A planner (Sonnet) handles complex multi-step reasoning. A memory writer consolidates every session. The HUD is the ambient display layer. All components communicate via WebSockets and a Redis event bus. The user never feels the complexity — they just speak, and Jarvis acts.

The intelligence stack — 4-tier LLM routing
Tier 0: Wake router Porcupine wake word detection → Silero VAD → Deepgram streaming STT starts. On-device, zero cloud. Classifies: is this for Jarvis? On-device
Tier 1: Fast classifier Claude Haiku 4.5 — receives transcript, assembles context from memory, routes to correct agent, handles simple Q&A directly. Target: <200ms response. API · $0.001/call
Tier 2: Reasoning engine Claude Sonnet 4.6 — complex multi-step tasks, code generation, research synthesis, nuanced judgment, tool orchestration chains. Used for ~20% of requests. API · $0.015/call
Tier 3: Extended thinking Claude Sonnet 4.6 with extended thinking — deep problem-solving, long-horizon planning, architecture decisions, essay/report generation. Used for <5%. API · $0.06/call
Tier 4: Local fallback Qwen2.5-7B-Instruct via Ollama — private data, offline mode, cost-sensitive tasks. Handles ~40% of queries when enabled. Zero cost, runs on GPU/NPU. Local · $0

6. Streaming TTSElevenLabs Flash v2.5 WebSocket. Receives text sentence-by-sentence. 75ms to first audio byte. Audio plays while more sentences generate.~75ms
7. Audio outputPyAudio / system audio. Streaming playback. Interrupt-capable: new wake word detection pauses TTS immediately.~10ms

5-layer memory architecture

Layer 1: Working memory

Current conversation context. Last 6–10 turns. Redis in-memory store. Expires after session ends. Token-compressed by Haiku if exceeds 4K tokens.

Layer 2: Episodic memory

What happened. Every session summarized into 3–5 sentences by Haiku. Stored as embeddings in Chroma vector DB. Retrieved by semantic similarity to current query. "Last week you were debugging the auth service."

Layer 3: Semantic memory

What you know and care about. Long-term facts: your job, family, preferences, goals. Structured JSON in SQLite + vectorized for retrieval. Updated explicitly or extracted automatically from conversations.

Layer 4: Procedural memory

How you work. Patterns Jarvis learns: "User prefers bullet-point summaries. Always cc Priya on project emails. Daily standup at 10am IST. Doesn't like unsolicited advice about diet."

Layer 5: Social graph

Who you know. Neo4j graph: Person nodes with relationships, contact info, last interaction, context. "Rahul: your college friend, works at Flipkart, last talked Mar 2025 about startup idea."

Layer 6: World model

Jarvis's model of your environment: home layout, devices, accounts, projects, files, recurring events. Enables "turn off the lights in my office" without explanation.

Memory pipeline — how memories are created and retrieved
Session end consolidationAfter each conversation: Haiku extracts key facts, generates episode summary, identifies new people/places/projects, updates SQLite + Chroma + Neo4j.
Retrieval at query timeParallel: vector search on episodic DB, SQL lookup for user profile, graph traversal for mentioned people. All assembled within 50ms before LLM call.
Contradiction resolutionIf new fact contradicts stored memory, Jarvis flags it: "I have you as vegetarian but you mentioned eating chicken — should I update this?" User confirms.
Forgetting policyEpisodic: keep 6 months rolling. Semantic: never auto-delete, only user-commanded. Social: archive after 1 year no interaction. User can wipe all data instantly.
Memory privacyAll memory stores local by default. Encrypted at rest (SQLCipher). Vector embeddings never leave device. Only anonymized summaries sent to cloud LLMs.

Security — Jarvis has god-mode, so security is critical

Critical risk: Jarvis can do anything

A compromised Jarvis can send emails, execute code, delete files, make purchases, and control your home. Every security layer below is mandatory, not optional.

Authentication

  • Voice biometric: speaker verification via Resemblyzer
  • PIN fallback for low-confidence voice auth
  • All API endpoints JWT-protected
  • Local network only by default (no external exposure)
  • Tailscale VPN for remote access

Credential security

  • All API keys in OS keychain (not .env in prod)
  • SQLCipher for all local databases
  • OAuth2 for Google/Spotify (no password storage)
  • Secrets rotation reminders built in
  • Vault mode: sensitive queries stay fully local

Action approval system

  • Confirmation required for: send email, delete files, execute code, make purchases
  • 30-second undo window after any action
  • Full audit log: every action timestamped + stored
  • Rollback available for reversible actions
  • Emergency stop: "Jarvis, abort everything"

Data privacy

  • Memory stores 100% local, never sent to cloud
  • Only anonymized/redacted text sent to Claude API
  • Screen capture: explicit opt-in, toggle with voice
  • GDPR-compliant: full data export + wipe on demand
  • Local LLM mode for maximum privacy

Prompt injection defense

  • Sanitize all web content before feeding to LLM
  • Detect injections: "Ignore previous instructions"
  • Tool execution always sandboxed
  • No arbitrary code from web sources
  • Input validation on all external data

Code sandbox

  • All code execution in Docker container
  • Network isolated by default
  • Filesystem access: temp dir only
  • CPU/memory limits enforced
  • Auto-kill after 30s timeout
